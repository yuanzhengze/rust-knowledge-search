## 二、项目目标与意义

### 目标

实现一个本地运行的命令行知识库搜索工具，让用户在不依赖任何外部服务的前提下，对本地的 Markdown / TXT 笔记进行高效的关键词检索和自然语言问答。

### 选题动机

1. 课程提示语中明确鼓励"本地知识库搜索系统"作为可探索方向，且是 Rust 不太常见的应用场景，能体现 Rust 在文件 I/O、字符串处理、数据结构、错误处理多方面的能力。
2. 现实中开发者笔记、课程资料分散在本地文件夹里，很需要一个不上传云端、隐私自留的检索工具。
3. 项目主体逻辑（扫描 / 解析 / 索引 / 排序）必须自己实现，能避免课程要求中"主要逻辑由 AI 自动生成"的问题；AI 仅作可选增强。

### 创新点

1. **零依赖中文分词**：CJK Unified Ideographs（U+4E00–U+9FFF）按字符 unigram 切分，ASCII 按字母数字累积，支持中英文混合查询而不引入 jieba 这类大字典依赖。
2. **行号对齐的 Markdown 清洗**：parser 在剥除 Markdown 标记时严格保持行数（代码围栏行替换为空行而非删除），让下游 chunker 的 `start_line` / `end_line` 能直接对应原始文件行号，便于未来精确高亮。
3. **POSIX 原子缓存写入**：storage 用"同目录临时文件 + `fs::rename`"方案，避免被中断时留半写文件；错误明确区分"语义级别索引未就绪"（`AppError::Index`）与"序列化失败"（`AppError::Serde`）。
4. **AI 可降级原则贯穿**：`SemanticEngine` 是 trait，`NoopEngine` 占位返回 `AppError::Semantic`，cli 层用 `match` 自动捕获并降级到候选片段展示。**整个项目可以在零网络、零 API Key 的环境下完整运行**。

## 四、核心数据结构

所有跨模块数据结构在 `docs/接口设计.md` §2 中固化。关键结构：

```rust
// scanner 输出 → parser 输入
pub struct DocumentMeta {
    pub path: PathBuf,
    pub file_name: String,
    pub file_size: u64,
    pub modified_time: Option<SystemTime>,
}

// parser 输出 → chunker 输入
pub struct Document {
    pub meta: DocumentMeta,
    pub title: Option<String>,
    pub content: String,  // Markdown 清洗后,行数与原文一致
}

// chunker 输出 → indexer/search 输入
pub struct DocumentChunk {
    pub id: String,           // <file_path>#L<start>-L<end>,稳定唯一
    pub file_path: PathBuf,
    pub title: Option<String>,
    pub content: String,
    pub start_line: usize,    // 1-based
    pub end_line: usize,
}

// search 输出
pub struct SearchResult {
    pub chunk_id: String,
    pub file_path: PathBuf,
    pub snippet: String,
    pub score: f64,
    pub matched_terms: Vec<String>,
}

// indexer 核心结构
pub struct InvertedIndex {
    pub terms: HashMap<String, Vec<String>>,                // term → chunk_ids
    pub term_freq: HashMap<String, HashMap<String, u32>>,   // term → (chunk_id → 频次)
}

// 统一错误类型
#[derive(Debug, thiserror::Error)]
pub enum AppError {
    #[error("IO error: {0}")]
    Io(#[from] std::io::Error),
    #[error("invalid path: {0}")]
    InvalidPath(String),
    #[error("parse error: {0}")]
    Parse(String),
    #[error("index error: {0}")]
    Index(String),
    #[error("semantic error: {0}")]
    Semantic(String),
    #[error("serde error: {0}")]
    Serde(#[from] serde_json::Error),
}

pub type AppResult<T> = Result<T, AppError>;
```

## 五、Rust 核心特性使用

### 1. 所有权与借用

- `&Path` / `&str` 在调用链上传递，避免不必要的克隆。例：`scan_documents(root: &Path)`、`search(index: &InvertedIndex, chunks: &[DocumentChunk], query: &str, top_k: usize)`。
- `InvertedIndex` 内部 `HashMap` 持有 `String` 键拥有所有权，跨函数边界传递时通过引用借出。
- 集成测试中通过 RAII 守卫实现 `TempDir`，`Drop` 自动清理临时目录，无需手动管理。

### 2. `Result` 与错误处理

- 所有可失败函数返回 `AppResult<T>`，没有任何 `unwrap()` / `expect()` 出现在产品代码（仅测试代码使用）。
- 用 `#[from]` 让 `std::io::Error` 与 `serde_json::Error` 自动转换为 `AppError`，配合 `?` 操作符让错误传播简洁。
- cli 层用 `match` 模式匹配区分 `Err(AppError::Semantic(_))`（降级）和其他错误（冒泡），是"可恢复错误"与"不可恢复错误"的精细分离。

### 3. trait 与多态

- `SemanticEngine` 定义统一接口（`rank` + `answer`），`NoopEngine` 是默认占位实现。未来真实实现（如 OpenAI embedding）只需 `impl SemanticEngine for OpenAIEngine`，cli 层无需任何改动。
- `serde::Serialize` / `serde::Deserialize` 派生让所有数据结构自动获得 JSON 序列化能力。
- `thiserror::Error` 派生让 `AppError` 自动实现 `std::error::Error` 与 `Display`。

### 4. 模块系统与可见性

- 严格按 `agent.md` §10 划分负责人，每个模块一个文件，公共类型在 `lib.rs` 顶层 re-export。
- `indexer::tokenize` 用 `pub(crate)` 暴露给同 crate 的 `search` 模块复用，但不进入对外公共 API，避免污染稳定接口。
- `IndexSnapshot` 公开声明便于后续调试工具直接反序列化。

### 5. 集合数据结构

- `HashMap<String, Vec<String>>` 实现倒排表（term → chunk_ids 列表）。
- `HashMap<String, HashMap<String, u32>>` 嵌套结构记录词频用于评分。
- `HashSet<&str>` 在 `InvertedIndex::build` 中做"单 chunk 内 term 去重"。
- `BTreeSet<String>` 在 search 输出 `matched_terms` 时保证顺序稳定。

### 6. 派生宏与零开销抽象

- `#[derive(Debug, Clone, Serialize, Deserialize)]` 大量使用，让数据结构自动具备调试输出、克隆与持久化能力。
- 所有评分排序逻辑都是纯函数，输入 `&InvertedIndex + &[DocumentChunk] + &str + usize` → 输出 `Vec<SearchResult>`，便于单元测试。

## 六、各模块设计要点

### scanner（谭张锐）

- 用 `walkdir::WalkDir::filter_entry` 在遍历期就剪枝隐藏文件、`target/`，避免遍历整个深目录后再过滤。
- 对 `walkdir::Error` 中的 IO 错误透传到 `AppError::Io`，目录权限问题不被静默吞掉。
- 单测用零依赖 `TempDir` 构造各种边界场景（缺失路径、文件路径、隐藏文件、不支持扩展名、`target/` 子目录）。

### parser（陈文涛）

- 用 `String::from_utf8` 显式校验，失败时构造 `AppError::Parse` 携带首个非法字节的偏移。
- Markdown 清洗的关键发现：**`(?m)^\s*#{1,6}\s+` 中的 `\s` 默认包含 `\n`，会跨行吞换行符**，让行数从 12 变成 10。修复方法是把行首前导 regex 中的 `\s` 改成 `[ \t]`。这一行为通过 `parse_markdown_should_preserve_line_count` 测试固化，防止回归。
- 围栏代码块行替换为空行，但围栏内容保留，让代码也能被索引到。

### chunker（陈文涛）

- 按行累积 + `max_chars` 阈值切片，单行超过阈值时该行单独成片，不在行内继续切，保留人类可读边界。
- chunk ID 形式 `<file_path>#L<start>-L<end>`，1-based 行号，稳定且唯一。
- chunks 之间行号连续不重叠：`chunks[i+1].start_line == chunks[i].end_line + 1`，便于回溯原文。
- `max_chars == 0` 自动 clamp 到 1 防止死循环。

### indexer（邱俊杰）

- **中英文混合分词** 关键决策：
  - ASCII 字母数字 + `_` 累积成 token，统一小写化。
  - CJK Unified Ideographs（U+4E00–U+9FFF）按字符 unigram 切分。
  - 其他字符（标点、空白、其他脚本）作为分隔符。
  - 这是知识库场景下的轻量做法，在没有任何分词依赖的前提下做到中英文都可搜。
- `tokenize` 用 `pub(crate)` 暴露给 search 复用，避免索引侧和查询侧规则漂移。
- 单 chunk 内 `terms` 去重，但 `term_freq` 累计真实频次。

### search（邱俊杰）

- 词频累加评分（MVP 实现），同分按 `chunk_id` 升序稳定排序，让单测可复现，未来 TF-IDF 升级只需改评分函数不动排序逻辑。
- 空查询 / 仅空白 / 仅标点 / `top_k == 0` 全部返回 `AppError::Index`。
- snippet 按 char 截断到 120（不按 byte，UTF-8 安全），上限通过常量 `SNIPPET_MAX_CHARS` 集中配置。
- `matched_terms` 用 `BTreeSet` 去重并保证顺序稳定。

### storage（袁正泽）

- 写入流程：序列化到内存 → 创建临时文件 `<path>.tmp.<pid>.<nanos>` → BufWriter 写入 + flush + sync_all → `fs::rename` 替换目标。
- 同设备 rename 是 POSIX 原子操作，被中断不会留下半写的目标文件。
- 缺失文件返回 `AppError::Index`（语义级"索引未就绪"），而非 `AppError::Io`（底层"文件不存在"），让 cli 层能给出对用户友好的提示。
- JSON 损坏走 `#[from] serde_json::Error` 自动转 `AppError::Serde`。

### semantic（袁正泽）

- `SemanticEngine` trait 设计为可替换：未来可以换成 OpenAI embedding、Anthropic Claude、本地 candle 模型，cli 层完全无感。
- `NoopEngine` 占位实现两个方法都返回 `AppError::Semantic("semantic engine disabled")`，并把 message 抽出为常量 `DISABLED_MSG` 供 cli 模式匹配复用。
- 真实 API Key / Base URL / 模型 ID 全部从环境变量读取（见 `.env.example`），代码里没有硬编码任何密钥。

### cli（黄开轩）

- 业务逻辑拆出 `*_at(cache_path: &Path, ...)` 函数，让集成测试可以传临时缓存路径，不污染 cwd。
- 缓存路径解析：`KNOWLEDGE_INDEX_PATH` 环境变量 > 默认 `.knowledge_index.json`（已在 `.gitignore`）。`cache_path_from_env` 是纯函数，单测无需碰全局 env 状态。
- ask 命令用 `match` 拆三路：`Ok(answer)` 走 AI、`Err(AppError::Semantic(_))` 走降级、其他错误冒泡。

## 九、遇到的问题与解决方案

### 1. Markdown 行首前导 regex 跨行吞换行

- **问题**：`(?m)^\s*#{1,6}\s+` 在 multiline 模式下，`\s*` 因为包含 `\n`，会向后贪婪匹配到下一行的内容，导致清洗后行数从 12 变成 10。
- **影响**：破坏"清洗后行数 = 原文行数"的不变量，下游 chunker 报告的行号无法回溯到原始文件。
- **修复**：把行首前导 regex 中的 `\s` 改成 `[ \t]`，限制为水平空白。`HEADING` / `LIST` / `BLOCKQUOTE` 三个 regex 同步修改。
- **预防**：增加测试 `parse_markdown_should_preserve_line_count`，断言清洗前后行数一致。

### 2. 跨设备 rename 不原子

- **问题**：最初考虑用系统临时目录写入再 rename 到目标位置，但跨文件系统时 `fs::rename` 会退化为"copy + delete"，不原子。
- **修复**：临时文件落在目标同目录下（`<path>.tmp.<pid>.<nanos>`），保证 rename 在同一文件系统内执行。

### 3. NoopEngine 错误信号要让 cli 能识别

- **问题**：cli 的 `ask` 命令需要区分"AI 没配置"（降级）和"AI 调用失败"（报错），但 `AppError::Semantic(String)` 只有 message。
- **修复**：把 disabled message 抽出为 `semantic::DISABLED_MSG` 常量，cli 层可以 `msg.contains("disabled")` 做精确判断（虽然当前是统一降级，但留下了扩展空间）。

### 4. shell 引号陷阱

- **问题**：早期开发中用 `git commit -m "..."` 提交，message 含反引号 `` ` `` 和尖括号 `<>` 时被 zsh 当成命令替换 / 重定向，commit 失败甚至产生奇怪的执行错误。
- **修复**：改用 `git commit --file=/tmp/commit-xxx.txt` 从文件读取 message，shell 完全不参与解析。文档中固化这个最佳实践。
