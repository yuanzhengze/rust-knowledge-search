# 演示视频脚本

本文档用于准备期末作业 5 分钟以内的功能演示视频。**MVP 已完成**，所有命令在 `examples/sample_notes/` 上均已 smoke test 通过，本脚本中"实际输出示例"区块直接来自真实运行结果。

## 1. 演示目标

演示需要让老师快速看到：

- 项目是一个完整可运行的 Rust 工具。
- 项目不是简单 API 包装，而是包含文件扫描、解析、索引、搜索等核心逻辑。
- 项目体现了 Rust 的模块化、错误处理、数据结构和工程实践。
- 小组成员分工明确。
- 项目具备一定创新性，支持本地知识库检索和可选 AI 语义增强；AI 不可用时不会失败。

## 2. 视频总时长

建议控制在 4 分 30 秒到 5 分钟之间。

## 3. 演示流程

### 0:00 到 0:40：项目背景与目标

讲解要点：

- 我们实现的是基于 Rust 的本地知识库语义搜索系统。
- 用户可以把课程笔记、Markdown 文档、TXT 资料放到本地目录中。
- 系统会自动扫描、解析并建立索引。
- 用户可以通过关键词或自然语言问题检索资料。
- AI 功能只是增强，核心检索流程由 Rust 实现，没有 API Key 也能跑。

示例讲稿：

```text
大家好，我们小组的 Rust 期末项目是"基于 Rust 的本地知识库语义搜索系统"。
这个项目面向本地资料检索场景，用户可以指定一个资料目录，系统会扫描
Markdown 和 TXT 文档，构建索引，并支持关键词搜索和自然语言问答。
我们重点使用 Rust 实现文件处理、索引构建、搜索排序和错误处理，
AI 只作为增强能力，没有 API Key 也能完整使用基础功能。
```

### 0:40 到 1:30：项目结构与成员分工

展示内容：

- 打开项目目录（VS Code 或终端 `tree`）。
- 展示 `src/` 模块结构。
- 展示 `README.md` 中的成员分工表格。

讲解要点（按数据流顺序介绍）：

- `scanner`（谭张锐）：递归扫描目录，过滤 `.md` / `.txt`。
- `parser`（陈文涛）：解析文档，清洗 Markdown，提取标题。
- `chunker`（陈文涛）：把长文档切成片段，保留行号范围。
- `indexer`（邱俊杰）：分词，构建倒排索引。
- `search`（邱俊杰）：候选合并、词频评分、Top K。
- `storage`（袁正泽）：原子写入 JSON 缓存。
- `semantic`（袁正泽）：可选语义增强（trait 设计 + NoopEngine 占位）。
- `cli`（黄开轩）：clap 子命令分发与结果渲染。
- 错误统一走 `error::AppError` + `AppResult<T>`。

可以快速展示 `cargo test` 的结果证明覆盖度：

```bash
cargo test 2>&1 | grep "test result"
```

实际输出：

```text
test result: ok. 65 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out
test result: ok. 3 passed; 0 failed; 0 ignored; 0 measured; 0 filtered out
```

示例讲稿：

```text
项目采用模块化设计，按照数据流分成 scanner、parser、chunker、indexer、
search、storage、semantic 和 cli 八个模块，每位组员负责一到两个模块。
模块之间通过 docs/接口设计.md 里固定的数据结构耦合，内部实现可以独立演化。
我们一共写了 65 个单元测试和 3 个集成测试，全部通过 cargo clippy 严格检查。
```

### 1:30 到 2:30：建立索引

演示命令：

```bash
KNOWLEDGE_INDEX_PATH=/tmp/demo_idx.json cargo run --quiet -- index ./examples/sample_notes
```

实际输出（已验证）：

```text
扫描到 4 个文档
切分得到 9 个片段
索引共 327 个关键词
索引已保存到 /tmp/demo_idx.json
```

讲解要点：

- 程序递归扫描 `examples/sample_notes/`，找到 4 份样例文档（2 个 Markdown + 2 个 TXT）。
- 解析阶段保持 Markdown 行数与原文一致，让后续切片报告的行号对应原始文件。
- 切片按字符上限 200 + 行号边界，每片一个稳定 ID `<path>#L<start>-L<end>`。
- 索引侧实现了**中英文混合分词**：CJK 按字符 unigram，ASCII 按字母数字累积；不引入任何分词依赖。
- 索引通过 `serde_json` 写入临时文件再 `rename`，POSIX 原子，被中断也不会留半写。

示例讲稿：

```text
这里我们使用 examples/sample_notes 作为示例知识库。运行 index 命令后，
程序扫描出 4 份文档，切分成 9 个片段，构建出包含 327 个关键词的倒排索引，
并通过原子写入保存到本地 JSON 缓存。下次搜索时直接读缓存，不需要重新扫描。
```

### 2:30 到 3:30：关键词搜索

演示命令：

```bash
KNOWLEDGE_INDEX_PATH=/tmp/demo_idx.json cargo run --quiet -- search "Rust 所有权" --top 3
```

实际输出（已验证，节选）：

```text
找到 3 条结果:

[1] ./examples/sample_notes/rust_ownership.md#L1-L14  分数 39.00
    匹配关键词: rust, 所, 有, 权
    Rust 所有权笔记

    Rust 的所有权系统用于在没有垃圾回收器的情况下保证内存安全...

[2] ./examples/sample_notes/rust_ownership.md#L15-L23  分数 16.00
    匹配关键词: rust, 所, 有, 权
    当一个值被赋给另一个变量时，如果类型没有实现 Copy，所有权会发生移动...

[3] ./examples/sample_notes/error_handling.txt#L1-L10  分数 5.00
    匹配关键词: rust, 有, 权
    Rust 错误处理笔记...
```

讲解要点：

- 查询 "Rust 所有权" 经过同一套分词后变成 4 个 token，每个 token 在倒排表里取候选。
- 评分用词频累加：第 1 条命中 4 个 token 各多次，分最高 39.00；第 2 条命中相同 token 但频次少，16.00；第 3 条只匹配部分 token，5.00。
- 同分按 chunk_id 升序稳定排序，测试可复现。
- 每条结果显示来源路径 + 行号范围 + 分数 + 匹配关键词 + snippet（按 char 截断到 120 字符，UTF-8 安全）。

示例讲稿：

```text
搜索命令支持中英文混合查询。我们查询"Rust 所有权"，系统返回三条结果，
按相关度从高到低排序。每条结果给出文件路径、行号、分数和匹配的关键词，
方便用户快速定位原文。同分情况下我们按 chunk_id 稳定排序，保证测试可复现。
```

### 3:30 到 4:30：自然语言问答（AI 降级路径）

演示命令：

```bash
KNOWLEDGE_INDEX_PATH=/tmp/demo_idx.json cargo run --quiet -- ask "Rust 的所有权是什么?"
```

实际输出（已验证，节选）：

```text
[提示] AI 引擎不可用 (semantic engine disabled)。以下是检索到的相关片段:

[1] ./examples/sample_notes/rust_ownership.md#L1-L14  分数 39.00
    Rust 所有权笔记

    Rust 的所有权系统用于在没有垃圾回收器的情况下保证内存安全...

[2] ./examples/sample_notes/rust_ownership.md#L15-L23  分数 20.00
    当一个值被赋给另一个变量时...
```

讲解要点：

- `ask` 子命令先调 search 取 Top 5 候选片段，作为 AI 引擎的上下文。
- 我们设计了 `SemanticEngine` trait，实现方可以替换为 embedding API 或本地模型。
- 当前默认使用 `NoopEngine` 占位实现，所有 AI 方法返回 `AppError::Semantic("semantic engine disabled")`。
- cli 层用 `match` 模式匹配捕获这个错误，**自动降级**为展示候选片段，并给出明确提示。
- 这就是 `agent.md` 里"AI 可降级"原则的代码体现：**没有 API Key、没有网络，本地知识库检索完全可用**。

示例讲稿：

```text
ask 命令用于自然语言查询。系统先用关键词检索找到相关片段，再尝试调用语义引擎
生成总结回答。当前我们使用占位实现 NoopEngine，所以会走到降级路径，
直接展示检索到的片段。SemanticEngine 是一个 trait，未来可以替换为
embedding API 或者本地轻量模型，cli 层不需要改动。
这种设计保证了项目在没有 AI 服务的环境下仍然完整可用。
```

### 4:30 到 5:00：Rust 特性与总结

讲解要点：

- 使用 `struct` / `enum` 建模文档、索引、搜索结果与错误（如 `AppError`）。
- 使用 `Result` 与自定义错误类型 `AppError`（thiserror 派生）做错误传播；模块代码里没有 `unwrap` / `expect`。
- 使用 `HashMap` / `HashSet` / `BTreeSet` 实现倒排索引、词频统计与稳定去重。
- 使用 `trait`（`SemanticEngine`）定义可替换的语义后端。
- 使用模块系统（`mod` + `pub(crate)`）控制可见性，`indexer::tokenize` 通过 `pub(crate)` 给 search 复用但不暴露。
- 通过 `cargo fmt`、`cargo clippy --all-targets -- -D warnings`、`cargo test` 三道质量门把代码守住。

示例讲稿：

```text
总结来说，本项目体现了 Rust 的模块化工程实践、所有权和借用管理、
Result 错误处理、HashMap/HashSet 等数据结构使用、trait 抽象，
以及测试和工程规范。我们用 65 个单元测试加 3 个集成测试覆盖了所有模块，
通过 clippy 严格检查零警告。相比单纯调用 AI API，我们把 Rust 作为
核心检索系统的实现语言，AI 只是增强层。以上就是我们的项目演示。
```

## 4. 录制前检查清单

录制前确认：

- 当前在仓库根目录（`pwd` 显示 `.../rust-knowledge-search`）。
- `cargo build` 已通过，无 warnings。
- `cargo test` 显示 65 单测 + 3 集成测试全过。
- `examples/sample_notes/` 4 份文档存在。
- `KNOWLEDGE_INDEX_PATH` 设置到非项目目录（如 `/tmp/demo_idx.json`），避免污染当前目录。
- 终端字体足够大（推荐 16pt+），背景对比度高。
- 不展示 `.env`、API Key、隐私路径或无关窗口。
- 关闭 IDE 中可能弹出的更新通知 / 邮件提醒。


## 6. 录制小贴士

- 用 `--quiet` 抑制 `cargo run` 的编译进度输出，画面更干净。
- 第一次跑可能触发编译，建议**录制前先 `cargo build` 一遍预热**，演示时直接 `cargo run` 就秒出。
- 如果想对比"重建索引"和"复用缓存"两种状态，可以在 `index` 命令前 `rm -f /tmp/demo_idx.json` 一下，再展示 `search` 直接从缓存读取。
- 演示 `ask` 降级时，**不要解释"AI 没接通"是 bug**——这是设计上的"可降级"特性，强调"AI 是增强不是核心"。
