# 基于 Rust 的本地知识库语义搜索系统两周开发计划

## 1. 项目信息

项目名称：基于 Rust 的本地知识库语义搜索系统

项目目标：使用 Rust 实现一个本地知识库智能检索工具，支持扫描本地 Markdown、TXT 等文档，构建关键词索引，并提供命令行查询功能。在基础检索能力完成后，加入语义搜索或 AI 问答增强能力，使用户可以通过自然语言问题检索本地资料。

## 2. 项目定位

本项目不是简单调用 AI API，而是以 Rust 实现核心数据处理流程，包括文件扫描、文档解析、文本切片、索引构建、搜索排序、结果展示和错误处理。AI 相关能力作为增强模块，用于提升自然语言查询体验。

项目重点体现以下 Rust 工程能力：

- 模块化设计：按照扫描、解析、索引、搜索、语义增强、命令行交互等模块拆分。
- 所有权与借用：在文档内容、搜索索引和查询结果传递中合理管理数据所有权。
- 错误处理：使用 `Result` 和自定义错误类型处理文件读取、解析、网络请求等异常。
- 数据结构：使用 `HashMap`、`Vec`、`HashSet` 等结构实现倒排索引和排序。
- 并发处理：可选使用线程或异步任务加速文件扫描与文档解析。
- 测试与工程规范：提供单元测试、关键功能测试，并通过 `cargo fmt` 和 `cargo clippy`。

## 3. 核心功能范围

### 3.1 必须完成的基础功能

- 扫描指定目录下的 `.md` 和 `.txt` 文件。
- 读取并解析文档内容。
- 将长文档切分为较短文本片段。
- 建立关键词倒排索引。
- 支持命令行搜索。
- 返回相关文档片段、文件路径和匹配关键词。
- 提供基础排序能力，例如词频排序或 TF-IDF 排序。
- 提供 README、运行说明和测试用例。

### 3.2 优先完成的增强功能

- 搜索结果关键词高亮。
- 索引结果保存到本地缓存文件。
- 支持重新加载已有索引，避免每次搜索都重新扫描。
- 支持交互式查询模式。
- 支持并发扫描和解析文档。

### 3.3 可选创新功能

- 接入 embedding API 或本地轻量模型，实现语义检索。
- 支持自然语言问答命令，例如 `ask`。
- 根据检索到的文档片段生成简短回答。
- 支持 PDF 文档解析。
- 提供 TUI 终端界面。

## 9. Git 协作规范

建议采用以下分支方式：

- `main`：稳定版本，只合并已经测试过的功能。
- `feature/scanner`：文件扫描模块。
- `feature/parser`：文档解析模块。
- `feature/indexer`：索引模块。
- `feature/cli`：命令行交互模块。
- `feature/semantic`：语义增强模块。
- `feature/docs`：文档和报告素材。

提交信息建议：

```text
feat: add directory scanner
feat: implement document parser
feat: build inverted index
feat: add search command
fix: handle empty query error
test: add parser unit tests
docs: update usage guide
```

协作要求：

- 每位成员至少负责一个模块。
- 每个功能尽量通过独立分支开发。
- 合并前先运行 `cargo fmt` 和 `cargo clippy`。
- 避免一次提交包含多个无关功能。
- README 中记录成员分工。

## 11. 风险与应对

### 风险 1：AI 语义功能实现不稳定

应对方案：

- 将 AI 功能设计为可选增强。
- 优先保证关键词搜索完整可用。
- 没有 API Key 或网络失败时，回退到普通搜索。

### 风险 2：PDF 解析耗时较高

应对方案：

- PDF 支持作为可选功能。
- 基础版本只要求支持 Markdown 和 TXT。
- 报告中说明后续可扩展 PDF 解析。

### 风险 3：搜索排序效果不明显

应对方案：

- 先实现词频排序。
- 再根据时间允许实现 TF-IDF 或简化 BM25。
- 准备高质量示例文档用于演示。

### 风险 4：模块集成出现接口不一致

应对方案：

- 第 1 天先确定核心数据结构。
- 各模块围绕统一结构开发。
- 第 7 天进行第一轮完整集成，不拖到最后。

### 风险 5：小组成员贡献不均衡

应对方案：

- 明确每位成员负责模块。
